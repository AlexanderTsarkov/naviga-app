## Canvas doc (source of truth): S03 — NodeTable как сердце продукта (план документации)

### Контекст и мотивация

- **Цель продукта:** Naviga — embedded-first; FW + радио + домен → BLE bridge → mobile позже.
- **Проблема-антипример:** Meshtastic решает «сообщения», геопозиция вторична; тяжёлые пакеты/протокол (protobuf), сложные радио-настройки, требующие от пользователя знаний (SF/BW/CR). У нас наоборот: **простота UX**, а сложность — под капотом.
- **Смысл S03:** довести **NodeTable** до статуса реального «сердца» продукта, чтобы в **S04** можно было безопасно протянуть её в BLE/BT и начать строить mobile UI/логику на фактах из NodeTable.

---

### Итог S03 (Definition of Done)

1) **NodeTable «полная»**: покрывает максимум полей, включая **reserved** (даже если часть ещё не летает по радио).
2) **Ясная типология данных**: поля разнесены по группам и привязаны к пакетам/источникам.
3) **Таблица соответствий** для каждого поля:
   - хранение в RAM (NodeTable)
   - persistence на flash (если применимо; если ещё не сделано — имплементировать)
   - on-air wire формат (как именно летит / размер / packing)
   - (под S04) BLE snapshot формат/поля
4) **Минимальный footprint в эфире**: где возможно — битовая упаковка, дифференциация по «частоте изменения».
5) **Оценка трафика**: расчёт/модель для 5–10–20 нод, несколько профилей; подтверждение, что оно реально летает (или зафиксированы корректировки).
6) **Provisioning / OOTB автономный старт**: новую железку можно прошить, включить — и она **сразу работает без телефона**:
   - FW делает первичную настройку железа «как это нужно Навиге»
   - создаётся/обновляется NodeTable, пакеты ходят с учётом общего трафика
   - необходимые данные корректно персистятся
7) **Ревизия пользовательских настроек (радио)**: канал/частота, LoRa параметры (скорость), мощность передачи, профили (factory/default → current → previous) — оформлено и приведено к продуктовой логике.
8) **Ревизия пользовательских настроек ноды**: роли/типы (человек/собака/инфраструктура/репитер/машина) и связанные параметры (min_time, max_silence, min_shift и т.п.) — проверены, уточнены, документированы.
9) **Док-ориентированная готовность к реализации**: планирование закрыто, решения оформлены, канон обновлён, есть мастер-тикет на изготовление + саб-тикеты на реализацию/тесты.

### Список результатов-артефактов (что будет написано в доках)

#### A. NodeTable Field Map (главный документ S03)

Таблица, где каждая строка — поле NodeTable.

**Обязательные колонки (минимум):**

- `Group` (см. группы ниже)
- `Field` (имя)
- `Semantics` (что означает)
- `Source of truth` (само-измерение / из эфира / derived)
- `Update trigger` (periodic / event(rx) / rare / user)
- `NodeTable storage` (тип/байты/packing)
- `Persistence` (persisted / derived / prohibited + правило)
- `On-air packet` (тип пакета + как кодируется)
- `Wire footprint` (байты payload + "on-air bytes" с заголовком)
- `Coalesce/replace rule` (как обновлять при RX/локально)
- `Notes` (sentinel, диапазоны, компрессия, риски)

#### B. Packet → NodeTable mapping

Карта: какие пакеты (on-air) заполняют какие группы/поля NodeTable.

#### C. Traffic model

Док/таблица/скрипт расчёта:

- сколько пакетов каждого типа в минуту/секунду
- какой средний on-air размер
- оценка нагрузки на эфир при 5/10/20 нодах и 2–3 профилях
- где границы и какие degrade-правила включаются

#### D. "Ready for S04" BLE snapshot outline

Тонкий документ-черновик: какие поля NodeTable нужны mobile UI первыми (map/list/status), какие можно отложить. **In S03 planning we only ship the Gate doc;** full BLE snapshot outline is S04 after WIP→Canon→Code→Test.

### Группы NodeTable (каноническая декомпозиция)

#### 1) Node_Core_Pos

- lat/lon (+ минимальные флаги валидности, если нужны)

#### 2) Node_Core_Tail

- остальная полезная GNSS-информация (примерный перечень для уточнения):
  - fix_type / sats / hdop-ish / speed / course / altitude / time
  - (важно) `ref_core_seq16` — связь с конкретным Core_Pos

#### 3) Node_Operational

Операционные параметры, которые могут меняться в runtime:

- **TX power (обязательно)**
- текущий радио-профиль / режим
- параметры энергосбережения/ролей/таймингов (что реально есть в FW)

#### 4) Node_Informative

Редко меняющиеся / «паспортные»:

- fwVersionId / hwProfileId (нужно формализовать, убрать "заглушки")
- manufacturer / capabilities (если есть)

#### 5) Node_Self_exposed

Наблюдаемое "на приёме":

- RSSI/SNR/quality принятых пакетов
- last_rx_at, rx_count (если решим)

#### 6) Node_Derived

Производные поля:

- stale/age, moving/idle, quality score, etc.

### Ключевые уточнения, которые надо зафиксировать в S03 (для твоих комментариев)

#### Provisioning (настройка железа под нужды продукта)

- Что именно FW делает «первым делом» на новой железке (E220, GNSS, power, режимы, тайминги).
- Какие значения считаются **factory defaults** для Naviga (а не «как пришло с завода»).
- Какие параметры обязаны попасть в NodeTable/telemetry сразу после provisioning.

#### OOTB автономный старт без телефона

- Минимальный набор настроек, который должен существовать, чтобы нода жила автономно:
  - default radio profile
  - default node role/type (и параметры)
  - baseline cadence/traffic policy
- Что персистится, чтобы после ребута нода продолжала вести себя предсказуемо.

#### Ревизия пользовательских настроек — радио

- Набор радионастроек (канал/частота, LoRa скорость/параметры, txPower step и т.п.).
- В S03 **нет** создания пользовательских профилей (нет UI/протокола) — задача S03: довести **OOTB/default** профиль до полноты и телеметрии.
- TX power обязательно отражается в Node_Operational и летит в эфир (Operational packet).
- Скорость/LoRa параметры (UART E22/E220): дискретные значения (2.4kbps, 4.8kbps, …). В S03 фиксируем как raw enum/step.
- Под SPI-модули (S05–S06) — отдельный будущий пересмотр модели/единиц.

#### Ревизия пользовательских настроек — нода/роль

- Набор типов/ролей: человек / собака / инфраструктура / репитер / машина.
- Какие параметры реально участвуют в поведении (min_time, max_silence, min_shift и т.п.) и где они живут.
- Что можно менять без телефона и что требует mobile (в S04).

#### TX Power — что именно значит "протащить"

**Не просто поле.** Нужно описать:

1) **Где живёт TX power как настройка:**
   - часть **radio profile**
   - есть **factory/default profile (OOTB)**
   - пользовательские профили (UI/протокол) **пока не доступны** и **не входят в S03**
2) **Состояния/история:**
   - минимум для S03: `default profile` + `current profile` (как активный)
   - `previous profile` — полезно, но можно оставить **reserved** до появления UI/протокола
3) **Телеметрия:**
   - read-back из радио (что реально выставлено) попадает в Node_Operational
4) **On-air:**
   - **TX power нужно передавать в Node_Operational уже в S03** — чтобы другие ноды знали, на какой мощности сейчас вещает нода (в будущем для авто-настройки сети)

**Формат:**

- Для E22/E220 UART модулей: мощность задаётся шагами (например 21/24/27/30/33; 33 — только для T33D).
- Поэтому в S03 canonical: хранить **raw step** (как authoritative), а `dBm` (если нужен) — только как derived/optional.
- Пересмотр единиц/диапазонов ожидается при поддержке SPI-модулей (ориентир S05–S06).

#### FW/HW version — убрать "пустые" поля

- определить, что такое `fwVersionId`/`hwProfileId`: opaque uint16 (таблица соответствий вне протокола) или что-то иное.
- определить sentinel/unknown поведение.

#### GNSS Tail — список и бюджет

- Сначала определить **GNSS HW profile** (режим настройки GNSS как часть provisioning) и получить список реально выдаваемых полей/вариантов.
- После этого выбрать:
  - must-have для S04 UI
  - optional / reserved
- Зафиксировать бюджет и degrade-политику.

#### Persistence — что реально persistится

- В S03, помимо `seq16`, должна переживать ребут **сама NodeTable** (минимум: ключевые поля + timestamps/age, по политике persisted/derived/prohibited).
- Определить: что хранить полностью, что — частично, что — запрещено.
- Зафиксировать стратегию записи (batched/flush vs часто) и риски износа flash.
